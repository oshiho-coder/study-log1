# 模写サイト⑦：コーポレートサイト

## サイト概要
Web制作会社のコーポレートサイトを想定した模写制作。  
トップページにおいて、ファーストビュー・ニュース・事業内容・会社情報・フッターまでを一通り実装し、  
HTMLの意味構造とレイアウト設計を意識したコーディングを行いました。

## サイト種類
- コーポレートサイト（企業公式サイト）

## ページ構成
- トップページ（1ページ）
  - Header（ロゴ／グローバルナビ／お問い合わせ導線）
  - Main Visual
  - News
  - About
  - Business
  - Company
  - Footer

## 対応端末
- PC
- スマートフォン（レスポンシブ対応）

## 使用技術
- HTML5
- CSS3
- Flexbox
- Grid Layout
- レスポンシブ対応（メディアクエリ）

## 実装内容
- header / nav / main / section / footer を用いた意味構造を意識したマークアップ
- Flexbox によるヘッダー内レイアウト（左右配置・中央揃え）
- Grid による Business セクションの2カラム構成
- 列単位で要素をまとめることで、左列のみ段差をつけるレイアウトを実装
- 擬似要素を用いたテキスト前装飾の実装
- hover 時の挙動を意識したインタラクション設計
- レスポンシブ時に Grid の列指定を解除し、縦並びへ切り替え

## 講師からの指摘
- navタグの使用範囲が曖昧で、ナビゲーション以外の要素を含めていた
- h1タグがページ内に複数存在していた
- 高さ調整を line-height に依存しており、Flexboxで解決すべき箇所があった
- hover の指定対象（親要素／子要素）の理解が不十分だった

## 詰まった点
- Gridで左列のみ段差をつける方法が分からず、marginやnth-childに頼って迷走した
- レスポンシブ時に1列指定しても2列に見える原因が分からなかった
- hoverを親にかけたつもりでも、意図した範囲に反映されなかった
- navや見出しタグの意味と見た目を混同していた

## 学んだこと
- レイアウトで詰まったときは、並べたい単位が適切かを疑う必要がある
- Gridは要素ではなく、列・行という座標で考える
- 親で指定を変えても、子の指定が残っているとレイアウトは崩れる
- position指定は要素を動かすためではなく、基準点を作るためのもの
- HTMLは意味、CSSは見た目という役割分担を意識する重要性

## 今後の改善点
- 位置調整を目的としたクラス（例：left）の使用を避け、  
  要素の役割が分かる命名（modifier 等）に整理する
- header の高さを固定値に依存せず、  
  コンテンツ量に応じて崩れにくいレイアウト設計を検討する
- レスポンシブ対応をモバイルファーストの設計に組み直す
- hover / focus / active の状態設計を体系的に整理する
